## 1. Executive Summary

Build **VulcanBench v1** — a fully open-source, transparent, community-driven benchmarking platform for evaluating LLMs on realistic software-engineering tasks in large codebases.

**Core Deliverables (MVP):**
- Python CLI harness (`vulcanbench`) with standardized agent tools, Docker sandboxing, multi-metric evaluation, and full JSONL+HTML trace capture.
- 5–10 high-quality, fully validated seed tasks (realistic MVP scope; 50 is post-v1).
- Next.js 15 + FastAPI dashboard with live leaderboard, per-run trace viewer (replayable), and GitHub PR submission flow.
- Complete reproducibility: `vulcanbench replay --run-id X` works from any trace.
- Apache 2.0 licensed monorepo with excellent README, docs, and one-command local setup.

**Key Insights from Research (SWE-bench, SWE-bench-Live, Harbor, evalh, etc.):**
- Docker containerization + pinned images is non-negotiable for reproducibility (SWE-bench 2024+ moved fully to this).
- Task creation is the hardest and most expensive part — automated pipelines (RepoLaunch-style) are complex; start small with hand-curated seeds + clear contribution path.
- Test brittleness is real (OpenAI audit 2026 found 59% of SWE-bench Verified had design flaws). Mitigate with `FAIL_TO_PASS` + `PASS_TO_PASS` + multiple gold-patch runs + LLM-judge fallback.
- ARM64 (Apple Silicon) support must be first-class; avoid x86-only assumptions.
- Traces (JSONL of every LLM call, tool invocation, diff, log) are the source of truth — enable replay, audit, and community trust.
- Fullstack starters (Next.js 15 App Router + FastAPI + Postgres + Docker Compose) are mature and well-documented.

**Success Criteria (from spec, adjusted for realism):**
1. `vulcanbench --help` + all core commands work.
2. Any seed task produces complete trace + leaderboard entry.
3. `vulcanbench replay` reproduces the exact run.
4. Dashboard runs locally and shows real data from harness.
5. All code typed/linted; core harness has ≥80% test coverage.
6. README enables one-command local setup + example run + "how to add a task".
7. GitHub PR template + CI validation for new tasks.

**Out of Scope for v1 (per spec):** Enterprise private repos, advanced human review queue, bounty system, synthetic task generator, 50 tasks.

## 2. Architecture Overview (Confirmed)

```
VulcanBench/
├── harness/                  # Python 3.12 + Typer + Pydantic v2 CLI & engine
│   ├── cli.py
│   ├── agent/                # Tool interface (list_files, read, edit, run_cmd, search, etc.)
│   ├── sandbox/              # Docker launcher + resource limits
│   ├── evaluator/            # Multi-metric scorer (functional/quality/security/efficiency/human_like)
│   ├── tracer/               # JSONL capture + HTML replay generator
│   └── requirements.txt / pyproject.toml
├── sandbox/                  # Docker artifacts
│   ├── Dockerfile.base
│   └── task.Dockerfile.template
├── tasks/
│   └── v1/                   # Seed tasks only (5-10)
│       └── swe-001/          # metadata.json, repo_snapshot.tar.gz (Git LFS), issue.md, tests/, ground_truth.patch, verifier.py, expected_metrics.json
├── dashboard/                # Next.js 15 (App Router) + Tailwind + shadcn/ui
│   ├── app/                  # Pages: /, /leaderboard, /task/[id], /run/[id], /submit
│   └── api/                  # (optional thin layer; or co-located FastAPI)
├── backend/                  # FastAPI + SQLModel + PostgreSQL (or Supabase local)
│   └── api/                  # /runs, /leaderboard, /run/{id}, /feedback
├── scripts/                  # validate_tasks.py, import helpers, task scaffolding
├── docs/                     # SPEC.md (this plan + original), QUICKSTART.md, ARCHITECTURE.md, TASK_CONTRIBUTION.md
├── .github/
│   ├── workflows/            # CI (lint, test, task validation)
│   ├── ISSUE_TEMPLATE/
│   └── PULL_REQUEST_TEMPLATE/
├── docker-compose.yml        # Full local stack (harness dev + dashboard + db + MinIO for LFS)
├── pyproject.toml
├── README.md
└── LICENSE (Apache 2.0)
```

**Data Flow (one run):**
1. `vulcanbench run --task swe-001 --model openai:gpt-4o`
2. Harness resolves task → launches isolated Docker container (pre-mounted repo snapshot at base commit).
3. Agent loop (custom ReAct or OpenAI tool-calling compatible) uses standardized tools.
4. Every step (LLM request/response, tool call/obs, file edit as unified diff, stdout/stderr, test output) → JSONL trace.
5. On completion: apply patch, run verifier + tests + linters + security scans.
6. Compute scores → persist (local JSON + optional Postgres) → generate HTML replay.
7. Leaderboard aggregates across runs.

**Storage:**
- Repo snapshots: Git LFS (or MinIO/S3-compatible for dashboard).
- Traces: Local `./runs/<run-id>/trace.jsonl` + `replay.html`.
- DB (dashboard): Runs, scores, feedback (PostgreSQL via Docker or Supabase).

**Clarifying Decision Needed:** Database strategy for dashboard v1?

### Phase 1: Harness Core (Core MVP — Highest Priority) (5-7 days)
**Goal:** Fully functional `vulcanbench run` that executes an agent, captures everything, scores, and produces replayable artifacts.

#### 1.1 CLI Foundation (`harness/cli.py`)
- Typer app with commands:
  - `run --task <id> --model <provider:model> [--output-dir] [--max-steps]`
  - `leaderboard --format (markdown|json|csv)` (local file-based for v1)
  - `replay --run-id <id> [--output html|json]`
  - `validate-task <path>`
  - `list-tasks`
- Pydantic v2 settings (model providers, sandbox limits, paths).
- Rich output + progress (tables, spinners).

#### 1.2 Agent Tool Interface (`harness/agent/`)
Standardized tools (must be identical inside sandbox and for replay):
- `list_files(dir, recursive?)`
- `read_file(path, start_line?, limit?)`
- `search_code(query, glob?, semantic?)` — start with ripgrep + tree-sitter; add sentence-transformers vector index later.
- `edit_file(path, old_string, new_string)` — returns unified diff; strict match required.
- `run_command(cmd, cwd?, timeout?)` — captures stdout/stderr/exit.
- `run_tests()`, `run_lint()`, `run_build()`, `security_scan()` — thin wrappers over task-specific scripts.
- Optional: `git_*` helpers (status, diff, checkout — shallow only).

**Design:** Protocol / ABC + concrete `DockerToolExecutor` (execs via `docker exec`) and `LocalToolExecutor` (for dev/testing only — never for real evals).

**OpenAI-compatible layer:** Expose tools as OpenAI function calling schema so any model (or existing agent framework) can drive the loop.

#### 1.3 Sandbox (`harness/sandbox/`)
- `DockerSandbox` class:
  - Builds/runs from `sandbox/task.Dockerfile.template` + task-specific overrides.
  - Mounts: read-only repo snapshot at `/workspace/repo`, writable `/workspace/patch` area.
  - Resource limits (cpus, memory, pids — configurable per task category).
  - Network: none or egress-only allowlist (security).
  - Non-root user inside container.
  - Healthcheck + auto-cleanup.
- Base image: `sandbox/Dockerfile.base` (multi-stage, pinned digests, minimal common tooling: git, ripgrep, tree-sitter parsers for Python/TS/Go/Java, Python/Node/Go runtimes).
- Template uses BuildKit cache mounts + secrets where needed.
- Support for ARM64 + x86_64 (manifest lists).

**Critical:** Pin every base image by digest. Document how to add new language runtimes.

**Research Note:** Follow SWE-bench 2024+ containerization patterns + modern multi-stage security (distroless where possible for runtime, but agent sandboxes need shells/tooling — use hardened alpine or wolfi with explicit allowlists).

#### 1.4 Tracer (`harness/tracer/`)
- `TraceCollector`: append-only JSONL writer.
- Events: `llm_request`, `llm_response`, `tool_call`, `tool_observation`, `file_edit` (with before/after + unified diff), `command_exec`, `test_result`, `metric_computed`, `error`.
- Every event has: `ts`, `step`, `run_id`, `task_id`, `model`.
- Post-run: `generate_replay_html(trace.jsonl) → self-contained replay.html` (timeline, searchable logs, diff viewer using Monaco or simple HTML, "replay step-by-step" simulation).
- Also emit `summary.json` (final scores, token counts, wall time, cost estimate).

#### 1.5 Evaluator (`harness/evaluator/`)
Compute:
- `functional`: % of `FAIL_TO_PASS` now passing + `PASS_TO_PASS` still passing (parse test output via task `verifier.py` or regex adapters).
- `quality`: ruff/pylint/eslint + radon complexity + maintainability (via radon or similar).
- `security`: bandit (Python), npm-audit / gosec equivalents; fail on high/critical.
- `efficiency`: total tokens (prompt+completion), wall-clock seconds, # of tool calls / edits.
- `human_like`: LLM-as-judge ensemble (3 cheap judges: e.g., gpt-4o-mini or local) scoring patch + trajectory against rubric (clarity of reasoning, minimal diff, correct style).
- `total`: weighted average (tunable weights in config).

**Verifier contract per task:** `verifier.py` (or `run_verifier.sh`) that takes final workspace state + expected and returns structured JSON `{functional: 87.5, details: [...]}`.

**Gold-patch validation:** On task import, run gold patch 3×; all must produce identical `PASS_TO_PASS`/`FAIL_TO_PASS` results.

#### 1.6 Model Provider Abstraction
- Support: OpenAI, Anthropic (via their SDKs or LiteLLM), local (Ollama/OpenAI-compatible), and "mock" for testing.
- For v1: Start with LiteLLM or direct OpenAI + Anthropic clients; keep interface narrow.

**Decision Point:** Custom ReAct loop inside harness vs. "bring your own agent" (pass a script that speaks the tool protocol)?
- Recommendation: v1 ships both — (a) built-in minimal ReAct for quickstarts, (b) documented "external agent" mode where user supplies a process that reads/writes the standardized tool JSON over stdio or HTTP.

#### 1.7 Local Leaderboard & Run Storage
- `./runs/` layout: `<run-id>/trace.jsonl`, `summary.json`, `final.patch`, `replay.html`, `logs/`.
- `leaderboard` command: scans `runs/`, aggregates, pretty-prints (supports filters by task category, model family).

**Tests for harness (target ≥80% coverage on core):**
- Unit: tool schema validation, diff application, metric math.
- Integration: mock sandbox (no Docker) end-to-end run with fake agent.
- Docker smoke: one real task in CI (marked slow).

### Phase 2: Tasks & Validation System (3-5 days + ongoing)
**Reality Check:** Creating 50 high-quality, decontaminated, executable tasks is 3-6+ months of full-time work for a team (see SWE-bench papers). v1 ships a **task framework + 5-10 excellent seeds** + automated validation + clear contribution guide. The "50" becomes a stretch goal / v1.1 milestone.

**Task Structure (exact per spec):**
```
tasks/v1/swe-001/
├── metadata.json          # {id, category: "bugfix|feature|refactor|arch|perfsec", languages, difficulty, created, source, decontamination_notes}
├── repo_snapshot.tar.gz   # Git LFS — clean shallow clone at base_commit
├── issue.md               # Raw GitHub issue text (no hints, no hand-holding)
├── tests/                 # Public tests + hidden verifier logic
├── ground_truth.patch     # PR diff minus test changes (for verification only)
├── verifier.py            # Custom or generic test parser + metric extractor
└── expected_metrics.json  # Gold run baselines
```

**Seed Task Strategy (MVP):**
- 2 Python bug fixes (small-mid repos, e.g., from recent post-2025 issues in popular libs).
- 2 TypeScript/TSX feature adds.
- 1 Go refactor or security fix.
- Prioritize: single-language, <50k LOC, fast tests (<2min), good existing test coverage.
- All must pass gold-patch validation 3× on both x86 and ARM64.

**Tooling (`scripts/`):**
- `validate_task.py <path>`: schema check, LFS presence, gold-patch replay in temp container, test determinism, decontamination (git log check for post-cutoff dates).
- `scaffold_task.py --from-pr <url>` (advanced; stretch): starts the tedious parts.
- `import_oss_issues.py` skeleton (documented for contributors).

**Decontamination:** Manual review + automated check that issue/PR dates are after training cutoffs (per model family). Document the exact cutoff used for each seed.

**Git LFS + Storage:** Use Git LFS for snapshots. Dashboard/MinIO for hosted. Document how to pull LFS objects.

**Contribution Path:** `.github/PULL_REQUEST_TEMPLATE/task-submission.md` with checklist (validation passed locally, gold runs 3×, decontamination signed off, license compatible, etc.). CI job runs `validate_task.py` on new PRs.

## 4. Critical Risks & Mitigations

1. **Task Dataset Scale (Highest Risk)**
   - 50 tasks in v1 is unrealistic for a solo/small-team launch.
   - **Mitigation:** Explicitly scope v1 to 5-10 seeds + rock-solid framework + contribution pipeline. Make "50 tasks" a v1.1 goal with community help. Publish a "Task Creation Handbook".

2. **Test Flakiness & Brittleness**
   - Real-world issue (documented in 2026 SWE-bench audits).
   - **Mitigation:** Mandate 3× gold runs per seed. Use `verifier.py` with structured output. Provide LLM-judge fallback for ambiguous cases. Clearly label any known-flaky tests in metadata.

3. **Docker / ARM64 / Reproducibility Nightmares**
   - Cross-arch images, base image drift, long build times.
   - **Mitigation:** Multi-arch manifests from day 1. Heavy use of BuildKit cache. Document "how to debug a failing sandbox". Pre-build and publish common base images to GHCR.

4. **Agent Loop Quality**
   - Building a good ReAct/tool-use loop from scratch is hard.
   - **Mitigation:** v1 includes a minimal working loop + excellent docs + examples for integrating existing agents (OpenHands, Aider, Claude Code via custom driver). Prioritize the tool interface contract over the loop itself.

5. **Storage & Cost (Repo Snapshots)**
   - Large tarballs per task.
   - **Mitigation:** Git LFS + shallow clones. Offer "lite" task variants. MinIO for dashboard dev.

6. **LLM-as-Judge Cost & Bias**
   - `human_like` metric.
   - **Mitigation:** Make it optional/configurable. Use cheap models. Publish exact judge prompts + allow self-hosting judges.

## 7. Non-Functional Requirements (Enforced Throughout)

- **Type Safety:** Pydantic v2 everywhere in Python; strict TypeScript in frontend.
- **Linting:** Ruff (Python), ESLint + Prettier (TS/JS). Zero warnings in CI.
- **Testing:** pytest (harness) + Playwright/Jest (dashboard). Fast unit tests on every commit; slow Docker tests on main/PR only.
- **Security:** Non-root containers, minimal attack surface, no secrets in images, `.dockerignore` strict.
- **Reproducibility:** Every public artifact must be replayable with one command + pinned images.
- **Accessibility & DX:** Excellent error messages, progress indicators, copy-paste friendly commands.
